## 一、第一轮修复（commit `a6c4204`）

### 1. 🔴 ZIP Slip 路径穿越漏洞

| 字段 | 内容 |
|------|------|
| **文件** | `core/downloader.go:143` |
| **等级** | 🔴 Critical — 安全漏洞 |
| **问题** | `ExtractZip` 解压时未校验 ZIP 条目文件名，恶意包含 `../../Windows/System32/malware.exe` 的 ZIP 会将文件写入任意系统路径 |
| **根因** | `fpath := filepath.Join(dest, f.Name)` 无 `..` 越界校验 |
| **修复** | 解压前检查 `filepath.Clean(fpath)` 是否以 `filepath.Clean(dest)` 为前缀，非法路径立即拒绝 |

```go
fpath := filepath.Join(dest, f.Name)
// ZIP Slip 防护：确保解压路径不跳出 dest 目录
if !strings.HasPrefix(filepath.Clean(fpath), filepath.Clean(dest)+string(filepath.Separator)) && fpath != filepath.Clean(dest) {
    return fmt.Errorf("zip 条目路径非法（可能为路径穿越攻击）: %s", f.Name)
}
```

### 2. 🔴 `SHBrowseForFolderW` PIDL 内存泄漏

| 字段 | 内容 |
|------|------|
| **文件** | `gui/gui.go:419-422` |
| **等级** | 🔴 High — 内存泄漏 |
| **问题** | `SHBrowseForFolderW` 返回的 PIDL 通过 `CoTaskMemAlloc` 分配，调用方必须调用 `CoTaskMemFree` 释放；原代码未释放，每次打开文件夹选择框泄漏 ~1KB |

```go
cf := windows.NewLazySystemDLL("ole32.dll").NewProc("CoTaskMemFree")
pidl, _, _ := bw.Call(uintptr(unsafe.Pointer(&bi)))
// ...
gp.Call(pidl, uintptr(unsafe.Pointer(&pb[0])))
cf.Call(pidl) // 释放 Shell32 分配的 PIDL
```

### 3. 🔴 `SwitchJDK` 多重失败时回滚结果不一致

| 字段 | 内容 |
|------|------|
| **文件** | `core/switcher.go:42-65` |
| **等级** | 🔴 High — 逻辑缺陷 |
| **问题** | `WriteEnvVar("JAVA_HOME")` 失败触发回滚后，`RestoreEnvVars` 自身也失败时，代码错误地将 `result.Error` 追加 **"已自动回滚"**（实际未回滚），用户误以为环境变量已恢复 |
| **根因** | 4 处回滚逻辑各自独立实现，`restoreErr != nil` 分支写"回滚也失败"，`else` 分支写"已自动回滚"，但某些分支（如 Path 读取失败）的 `else` 缺失 |
| **修复** | 提取 `doRollback(backupFile string) error` 统一函数，每次调用后 `err == nil` 才写"已自动回滚" |

```go
func doRollback(backupFile string) error {
    return RestoreEnvVars(backupFile)
}

// 使用方
rollbackErr := doRollback(result.BackupFile)
if rollbackErr != nil {
    result.Error += fmt.Sprintf("（回滚失败: %v）", rollbackErr)
} else {
    result.Error += "（已自动回滚）"
}
```

### 4. 🔴 `CleanPath` 清洗规则过于激进，误删合法路径

| 字段 | 内容 |
|------|------|
| **文件** | `core/switcher.go:117-120` |
| **等级** | 🔴 High — 逻辑缺陷 |
| **问题** | 原逻辑用 `strings.Contains(lower, `\java\`)` / `\jdk\` / `\jre\` 通配匹配，会把系统其他工具（如 IDE 自带的 JRE、Android Studio 的 Java 路径等）误删，即使它们与旧 JDK 无关 |
| **根因** | 清洗策略基于路径关键词猜测，而非精确匹配旧 JDK 的绝对路径 |
| **修复** | 改为仅删除 `oldJdkDirLower` 精确前缀匹配（其自身及所有子目录）+ `%JAVA_HOME%\bin` 字面量 |

```go
shouldRemove := false
// 字面量 "%JAVA_HOME%\bin"
if lower == "%java_home%\\bin" {
    shouldRemove = true
}
// 旧 JDK home 的直接子目录
if oldJdkDir != "" {
    oldPrefix := oldJdkDirLower + string(filepath.Separator)
    if lower == oldJdkDirLower || strings.HasPrefix(lower, oldPrefix) {
        shouldRemove = true
    }
}
```

### 5. 🔴 `ExtractZip` 单层目录提升竞态（临时目录冲突）

| 字段 | 内容 |
|------|------|
| **文件** | `core/downloader.go:173-179` |
| **等级** | 🔴 High — 竞态条件 |
| **问题** | 解压后"单层目录提升"逻辑使用 `dest + "_tmp"` 作为临时路径；若两个 JDK 并发下载解压，不同 `dest` 的 `_tmp` 后缀相同，`os.Rename` 可能将 A 的目录重命名为 B 的 `_tmp`，导致数据混乱 |
| **根因** | 临时目录名未包含随机/唯一标识 |
| **修复** | 使用 `os.MkdirTemp(filepath.Dir(dest), ".jvs-extract-*")` 生成唯一临时目录名 |

```go
tmpDest, mkErr := os.MkdirTemp(filepath.Dir(dest), ".jvs-extract-*")
os.Rename(innerDir, tmpDest)
os.RemoveAll(dest)
os.Rename(tmpDest, dest)
```

### 12. 🔴 `ExtractZip` `rc`/`outFile` double-close + 句柄泄漏

| 字段 | 内容 |
|------|------|
| **文件** | `core/downloader.go:158-171` |
| **等级** | 🔴 Critical — 文件句柄泄漏 |
| **问题** | `f.Open()` 失败时（`rc == nil`）手动调 `rc.Close()` 无效果 → `outFile` 保持打开 → 函数返回 → `defer r.Close()` 只关 `zip.Reader`，不关 `outFile` → **文件句柄泄漏** |
| **根因** | 无 `defer` 统一管理文件生命周期 |
| **修复** | `rc, err := f.Open()` 后 `defer rc.Close()`；`os.OpenFile` 后 `defer outFile.Close()`；`io.Copy` 后不再手动 Close |

```go
rc, err := f.Open()
if err != nil {
    return err
}
defer rc.Close()

outFile, err := os.OpenFile(fpath, os.O_WRONLY|os.O_CREATE|os.O_TRUNC, f.Mode())
if err != nil {
    return err
}
defer outFile.Close()

if _, err = io.Copy(outFile, rc); err != nil {
    return err
}
```

### 15. 🟠 `BackupFilePath` 与 `BackupEnvVars` 路径分裂

| 字段 | 内容 |
|------|------|
| **文件** | `core/registry_ops.go:154-165` |
| **等级** | 🟠 High — 逻辑缺陷 |
| **问题** | `BackupFilePath()` 内部调 `MkdirAll` 创建目录；失败时降级到 `%TEMP%` 路径。但 `BackupEnvVars(bf)` 再次调 `MkdirAll(bf的目录)`，`%TEMP%` 通常已存在，两次 `MkdirAll` 行为不一致 → `result.BackupFile` 指向降级路径，实际写入的是另一条路径 |
| **根因** | 两个函数各自独立创建目录，路径计算重复 |
| **修复** | `BackupFilePath` 只负责生成路径字符串，删除所有 `MkdirAll`；`BackupEnvVars` 统一负责目录创建 + 文件写入，单一职责 |

```go
// BackupFilePath: 只生成路径，不创建目录
func BackupFilePath() string {
    appData := os.Getenv("APPDATA")
    // ...
    return filepath.Join(appData, "JVS", "backup", time.Now().Format("20060102_150405")+".reg")
}

// BackupEnvVars: 统一负责创建目录 + 写入
func BackupEnvVars(backupFile string) error {
    // ...
    if err := os.MkdirAll(filepath.Dir(backupFile), 0755); err != nil {
        return fmt.Errorf("创建备份目录失败: %w", err)
    }
    return os.WriteFile(backupFile, []byte(b.String()), 0644)
}
```
